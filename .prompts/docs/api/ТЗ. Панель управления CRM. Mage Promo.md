## Требования к запросам API

**Протокол:** Передача данных только по HTTPS с использованием TLS 1.2 или выше.  
**Архитектура**: Соответствие принципам REST (использование ресурсов в URL, например /api/v1/deals).  
**Формат данных:** Все данные в теле запроса должны передаваться в формате JSON (за исключением загрузки файлов).  
**Кодировка:** Строго UTF-8.  
**Headers**:  
Content-Type: application/json (для обычных данных).  
Content-Type: multipart/form-data (только для загрузки файлов).  
**Authorization**: Bearer \<token\> (для всех защищенных методов).

### Методы

#### Метод GET

В таблице указаны параметры запроса в JSON. Однако, для метода GET (по стандартам игнорирующий тело запроса body) параметры передаются через url-параметры ?param1=value1\&param2=value2…  
Если передаются сложные параметры \- массивы или объекты, они кодируются в url-строку.

#### Остальные методы

Данные запроса передаются штатно через body в формате JSON.

### Формат пути запроса

#### Требования к формированию путей запросов API

**Использование существительных**: Пути должны указывать на ресурсы, а не на действия.   
Правильно: POST /deals (создать сделку)  
Неправильно: POST /createDeal или /get\_all\_contacts  
**Использование глаголов**  
Если действие – это не изменение поля, а отдельный процесс. Например:  
POST /v1/users/123/reset-password  
**Множественное число**: Все ресурсы именуются во множественном числе для единообразия.  
Правильно: /users, /files, /tasks  
Неправильно: /user, /file  
**Версионность**: Путь должен начинаться с номера версии. Это позволит обновлять API, не ломая работу старых клиентов.  
Шаблон: /api/v1/{resource}  
**Регистр и символы**:  
Использовать только нижний регистр (lowercase).  
Вместо пробелов и нижнего подчеркивания использовать дефис (kebab-case), если название ресурса состоит из двух слов.  
Правильно: /crm-settings  
Неправильно: /crm\_settings или /crmSettings  
**Идентификаторы (ID)**: Для обращения к конкретному объекту ID передается в самом пути.  
Шаблон: /{resource}/{id} (например, /contacts/550)

#### Разделение ресурсов между кампаниями

В путь запроса, первым параметром, включается ID кампании (promo\_id в текущих таблицах). На уровне сервера, разделение ресурсов между кампаниями производится на усмотрение бэкенд-команды.

#### Формат строки запроса

/api/{version}/{campaign\_id}/{resource}/{object\_id}

## Требования к ответам

**Единая структура**: Ответ всегда должен быть JSON-объектом.  
**Именование полей:** snake\_case  
**HTTP-статусы**: Обязательное использование корректных кодов состояния:  
200 OK – успешный запрос.  
201 Created – объект создан успешно.  
4xx – ошибки на стороне клиента.  
5xx – ошибки на стороне сервера.  
**Обработка пустых данных**:  
Если список пуст, возвращать пустой массив \[\], а не null или ошибку 404\.  
Если объект не найден, возвращать 404 Not Found.  
**Формат дат**: Использование международного стандарта ISO 8601 (например, 2023-10-27T10:00:00Z).

## Ошибки запросов {#ошибки-запросов}

### Формат ответа

При ошибке, API возвращает ответ в следующем формате:

| {   "error": {     "code": "AUTH\_REQUIRED",     "message": "Для выполнения действия нужна авторизация.",     "details": \[    // Необязательная секция       {         "field": "email",         "issue": "Неверный формат почты"       },       {         "field": "password",         "issue": "Пароль слишком короткий"       }     \]   }   } |
| :---- |

### Перечень ошибок и их описание

| HTTP | Код в JSON (code) | Сообщение в JSON (message) | Описание ситуации |
| :---- | :---- | :---- | :---- |
| 400 | INVALID\_SYNTAX | Ошибка в структуре запроса. | Битый JSON или пропущены поля. |
| 401 | UNAUTHORIZED | Требуется авторизация. | Токен отсутствует или невалиден. |
| 403 | ACCESS\_DENIED | Доступ запрещен. | Нет прав на конкретный ресурс/действие. |
| 404 | RESOURCE\_NOT\_FOUND | Ресурс не найден. | Объект с таким ID не существует. |
| 405 | METHOD\_NOT\_ALLOWED | Метод не поддерживается. | Например, вызвали GET вместо POST. |
| 409 | ENTITY\_ALREADY\_EXISTS | Объект уже существует. | Конфликт (например, дубликат email). |
| 413 | FILE\_TOO\_LARGE | Файл слишком большой. | Превышен лимит загрузки. |
| 415 | UNSUPPORTED\_MEDIA\_TYPE | Неподдерживаемый тип данных. | Неверный Content-Type в заголовке. |
| 422 | VALIDATION\_ERROR | Ошибка валидации данных. | Данные корректны, но не логичны. |
| 429 | TOO\_MANY\_REQUESTS | Превышен лимит запросов. | Слишком много обращений к API. |
| 500 | INTERNAL\_SERVER\_ERROR | Внутренняя ошибка сервера. | Необработанное исключение в коде. |
| 502 | BAD\_GATEWAY | Плохой шлюз. | Ошибка связи между Nginx и бэкендом. |
| 503 | SERVICE\_UNAVAILABLE | Сервис временно недоступен. | Техработы или перегрузка сервера. |
| 504 | GATEWAY\_TIMEOUT | Время ожидания истекло. | Бэкенд не ответил вовремя. |

## Получение списков/таблиц {#получение-списков/таблиц}

Запрос должен содержать параметры пагинации, фильтрации, сортировки.  
Ответ должен содержать метаданные списка/таблицы, параметры пагинации, фильтрации, сортировки.  
Сортировка предусматривается только по одному столбцу.

| /api/1/{campaign\_id}/{resource} |
| :---- |
| GET |
| {                                  // Все поля необязательны.    “first”:0,                   // Номер страницы   “limit”:50,                   // Размер страницы. При отсутствии \- неограниченный вывод   “sort”: “id1”,               // ID столбца сортировки. При отсутствии \- без сортировки   “direction”:”asc”,        // Направление сортировки (asc, desc)   “search”:””,                // Строка быстрого поиска по данным полей записей   “filters”: \[                    // Набор фильтров      { “id”:”id1”, “value”:... }       …     \], } |
| {     “columns”: \[             // Описание столбцов. Может отсутствовать, если структура       {                            // статичная и шапка формируется на фронте.          “id”:”id1”,            // ID столбца         “title”: “Имя”,        // Название столбца. Необязательно         “type”: “string”      // Тип данных. Необязательно         “sortable”:true      // Разрешает сортировку по столбцу         “props”: {}            // Набор свойств, применяемых ко всем ячейкам столбца       }       …     \],     “rows”: \[                       // Ряды       {         “id”: “row1”               // ID ряда. Необязательный. Нужен для управления записью         “values”: {                // Список пар имя=значение записей           “id1”: “Значение1”,           “id2”: “Значение2”           …         },         “props”: {}                // Набор свойств, применяемых ко всем ячейкам строки       },       …     \],     “filters”: \[                    // Набор доступных фильтров      { “id”:”id1”, “name”:”Название 1”, “type”:”number”, “opts”:..., “value”:... }       …     \],     “search”:””,                 // Строка быстрого поиска     “meta”: {                    // Набор метаданных, связанных с пагинацией и сортировкой       "total\_count": 1450,       "first": 1,       "limit": 50,       “sort”: “id1”,       “direction”: “desc”     } } |
