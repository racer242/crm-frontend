# Авторизация NextJS + Битрикс

## Цель

Реализовать авторизацию на портале NextJS, где источником пользователей является 1С Битрикс. Битрикс выдаёт JWT, NextJS валидирует его самостоятельно без обращения к Битрикс на каждый запрос.

---

## Ключевые факты

- Все запросы к Битрикс идут **только с сервера NextJS**, никогда с браузера
- Токены хранятся исключительно в **httpOnly cookies**
- Middleware валидирует токен **локально** через JWT_SECRET, без сетевых запросов
- Данные пользователя для UI читаются **на сервере** в `layout.tsx` и передаются в Context, чтобы избежать мигания меню
- `JWT_SECRET` — один и тот же на стороне Битрикс и NextJS
- Переменные `BITRIX_API_URL`, `BITRIX_INTERNAL_SECRET`, `JWT_SECRET` — серверные, без префикса `NEXT_PUBLIC_`

---

## Стек

`Next.js 14+ App Router` · `TypeScript` · `jose` (JWT) · `React Context` (состояние авторизации)

---

## Этапы

### 1. Подготовка

- Установить пакет `jose`
- Создать `.env.local` с тремя переменными: `BITRIX_API_URL`, `BITRIX_INTERNAL_SECRET`, `JWT_SECRET`
- Зафиксировать структуру в `.env.example` без значений

---

### 2. Типы и константы

- Описать типы: `User`, `AuthTokens`, `LoginResponse`, `LoginCredentials`, `TokenPayload`
- В `TokenPayload` обязательно включить: `sub`, `login`, `email`, `name`, `avatar`, `role`, `groups`, `iat`, `exp`, `type`
- Вынести в константы: ключи cookies, время жизни токенов, списки защищённых и гостевых роутов

---

### 3. Инфраструктурные сервисы

Создать три сервиса:

**bitrixClient** — HTTP клиент для запросов к Битрикс. Всегда передаёт `X-Internal-Secret` в заголовке. Экспортирует класс ошибки `BitrixApiError` со статусом.

**tokenService** — работа с JWT через `jose`. Методы: верификация токена, декодирование без верификации, проверка что токен истекает в ближайшие 5 минут.

**cookieService** — установка и удаление auth cookies. Работает только в Route Handlers и Server Components (не в middleware). `access_token` и `refresh_token` — httpOnly. `user_data` — не httpOnly, доступен клиенту для инициализации Context.

---

### 4. API Routes

Три endpoint-а:

**POST `/api/auth/login`** — принимает логин/пароль, отправляет на Битрикс, при успехе устанавливает три cookie, возвращает объект `user` в body.

**POST `/api/auth/refresh`** — берёт `refresh_token` из cookie, отправляет на Битрикс, обновляет cookie с новыми токенами. При ошибке — удаляет все cookies и возвращает 401.

**POST `/api/auth/logout`** — берёт `access_token` из cookie, уведомляет Битрикс, удаляет все cookies. Завершается успехом даже если Битрикс недоступен.

---

### 5. Middleware

Файл `src/middleware.ts`. Логика:

1. Определить тип роута: защищённый, гостевой, остальные
2. Проверить `access_token` локально через `tokenService`
3. Если токен истекает — попробовать refresh через внутренний fetch на `/api/auth/refresh`, перенести новые cookies в ответ
4. Защищённый роут без сессии → редирект на `/login?returnUrl=...`
5. Гостевой роут с сессией → редирект на `/dashboard`

**Важно:** `cookieService` использует `next/headers` и несовместим с Edge Runtime. В middleware работать только с `request.cookies` напрямую.

---

### 6. Серверный хелпер

Функция `getServerUser()` — читает `access_token` из cookies через `next/headers`, верифицирует локально, возвращает `TokenPayload` или `null`. Используется в Server Components.

---

### 7. Auth Context

**`AuthProvider`** — клиентский компонент. Принимает `initialUser: User | null` пропом с сервера. Хранит состояние: `user`, `isLoading`, `error`. Реализует методы `login`, `logout`, `clearError`.

**`useAuth`** — хук для доступа к контексту в клиентских компонентах. Бросает ошибку если вызван вне провайдера.

---

### 8. Layout

`src/app/layout.tsx` — асинхронный Server Component. Вызывает `getServerUser()`, формирует объект `User` из `TokenPayload`, передаёт в `AuthProvider` как `initialUser`. Это исключает мигание меню при загрузке.

---

### 9. Меню

Клиентский компонент. Получает данные через `useAuth()`. Отображает аватар, имя, роль пользователя и кнопку выхода. Данные доступны сразу при первом рендере — мигания нет.

---

### 10. Страница логина

Клиентский компонент по пути `/login`. Использует `useAuth()` для вызова `login()`. После успешного входа делает `router.push(returnUrl)` и `router.refresh()` для обновления Server Components.

---

## Порядок проверки после реализации

```
1. Успешный логин → появились три cookie, редирект в дашборд
2. Неверные данные → ошибка на форме, cookies не установлены
3. Открыть /dashboard без авторизации → редирект на /login
4. Открыть /login с авторизацией → редирект на /dashboard
5. Logout → cookies удалены, редирект на /login
6. Меню отображает имя и аватар без мигания при загрузке
```
